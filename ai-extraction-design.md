# AI Extraction Design — Senus PLC Board Report

## 1. Pipeline Shape: 4 AI Calls Per New Document

Per the "one prompt per record type" decision, processing a newly ingested document runs up to 4 focused AI calls, not one large one:

1. **Metric extraction** — pulls raw financial figures
2. **Risk extraction** — pulls risk factor disclosures (only run on documents that contain a risk section)
3. **Event extraction** — pulls one-off corporate announcements (only run on press releases/announcements)
4. **Risk status diffing** — compares this document's extracted risks against the previously stored set (only run after step 2, and only if prior risk records exist to compare against)

Document metadata (`doc_type`, `period_label`, `consolidation_basis`, `audited`) is set programmatically from the Bucket 2 inventory, not asked of the LLM — the LLM only ever produces the `payload` content, never the surrounding classification. This keeps the boundary clean: humans/code decide *what kind of document this is*, AI decides *what's in it*.

## 2. Metric Extraction Prompt

**Scope:** raw figures only — revenue, cost of sales, gross profit, operating loss, cash balance, debtors, creditors, etc. Explicitly does NOT ask the LLM to calculate ratios (see §4 below for why).

```
You are extracting raw financial figures from a Senus PLC source document.
Return ONLY a JSON array. Each object must have exactly these fields:
{
  "metric_name": string (e.g. "revenue", "cost_of_sales", "gross_profit", "cash_and_equivalents"),
  "value": number (no currency symbols, no commas),
  "unit": "EUR" | "GBP",
  "comparative_value": number or null,
  "comparative_period": string or null (e.g. "H1 FY2025")
}
Extract only figures explicitly stated in the document. Do not calculate
or infer any ratios, percentages, or derived figures — extract raw line
items exactly as reported.
If a figure is not present in this document, omit it entirely rather than
guessing or inserting a null value.
```

## 3. Risk Extraction Prompt

**Scope:** one object per disclosed risk, in the order they appear in the source (this preserves the Directors' own most-material-first ordering without asking the AI to invent a numeric score).

```
You are extracting risk factor disclosures from a Senus PLC source document.
Return ONLY a JSON array, preserving the exact order risks appear in the
source document within each category. Each object must have exactly:
{
  "category": string (use the document's own category headings, e.g. "Corporate", "Technology and IP"),
  "title": string (short, e.g. "Cyber security threats to customer data"),
  "summary": string (1-2 sentences, paraphrased in plain English — do not
    copy sentences verbatim from the source)
}
Only extract risks explicitly described as risk factors or as risks/
uncertainties affecting the Company. Do not infer risks that aren't stated.
```

Materiality rank is then assigned by code as the array index within each category — not asked of the LLM, since "which order did these appear in" is a mechanical fact, not an interpretive one.

## 4. Why Derived Metrics Are Computed in Code, Not by the LLM

**Decision:** Gross margin, EBITDA margin, and other ratios are calculated by a small deterministic function reading the raw `metric` rows — never generated by asking the LLM to do arithmetic.

**Why:** LLMs are not reliable calculators — arithmetic errors, however rare, are an unnecessary risk for numbers that feed a board-level financial report. Division is not an interpretive task the way "what does this paragraph mean" is; it has exactly one correct answer, so it belongs in code, not a prompt. This isn't a contradiction of "AI-led, not rule-based" — that principle was about *interpreting messy, variably-formatted source material*, not about *doing math*. The right tool for arithmetic is a function; the right tool for reading a differently-formatted balance sheet each time is an LLM.

## 5. Risk Status Diffing — AI-Led Comparison, Not String Matching

**Decision:** When a document introduces new risk extractions, comparing them against the previously stored set to set `status` (New / Updated / Unchanged) is itself a 4th AI call, not a fuzzy string-matching algorithm.

```
You will be given two lists of risk disclosures: PREVIOUS (from an earlier
document) and CURRENT (just extracted from a new document). For each item
in CURRENT, decide:
- "Unchanged" — clearly the same risk as one in PREVIOUS, same substance
- "Updated" — clearly the same risk as one in PREVIOUS, but meaningfully
  reworded, expanded, or changed in emphasis
- "New" — no clear match in PREVIOUS

Return ONLY a JSON array matching CURRENT's order:
{ "title": string, "status": "New" | "Updated" | "Unchanged", "matched_previous_title": string or null }

Risks are often reworded between documents without changing in substance —
judge by meaning, not exact wording.
```

**Why AI-led rather than string similarity matching:** this is consistent with the same reasoning used for extraction itself — two risk descriptions can be substantively identical while being reworded (the HY report literally does this, referring back to the Information Document's risks "in substance" without repeating them verbatim). A string-matching or embedding-similarity threshold would need constant tuning and could misfire on exactly the paraphrasing Senus's own disclosures already do. An LLM judging semantic equivalence is the same tool already trusted to read the documents in the first place.

## 6. Validation Approach

The brief explicitly asks how outputs were validated. The concrete answer: the source documents were read manually during Buckets 1–2 of this project, and several figures were captured and cross-checked by hand *before* any extraction code was written — e.g. FY2025 revenue (€836,991), H1 FY2026 revenue (€354,813), FY2025 gross margin (77.5%), cash balance at Dec 2025 (€735,189). These hand-verified figures become a small **golden test set**: once the extraction pipeline runs, its output is checked against these known-correct values first, before trusting it on anything not manually checked. A mismatch here means a prompt or parsing bug, not a data problem, and is the first thing to fix.

## 7. Raw Metric Taxonomy

Drawn from what's actually disclosed across the in-scope source documents (Bucket 2), grouped by the metric categories from Bucket 1. This is the fixed vocabulary the metric extraction prompt is told to recognise — new line items Senus discloses in future reports can still be added here without touching the schema itself.

**Growth & Revenue**
`revenue`, `enterprise_customer_count`, `independent_customer_count`, `rd_customer_count`, `acv_enterprise_soil`, `acv_enterprise_era`, `acv_enterprise_terrain`, `pipeline_bookings_closed_value`, `pipeline_open_value`, `pipeline_deals_closed_count`

**Profitability**
`cost_of_sales`, `gross_profit`, `administrative_expenses`, `other_operating_income`, `operating_profit_loss`, `profit_loss_before_tax`, `profit_loss_after_tax`

**Cash & Liquidity**
`cash_and_equivalents`, `cash_at_period_start`, `net_cash_operating_activities`, `net_cash_investing_activities`, `net_cash_financing_activities`, `debtors`, `creditors_due_within_one_year`, `creditors_due_after_one_year`, `depreciation_amortization`

**Solvency & Leverage / Balance Sheet**
`net_assets_liabilities`, `called_up_share_capital`, `share_premium`, `retained_earnings`, `goodwill`, `development_costs`, `tangible_assets`, `contingent_consideration`

**Returns / Market** (sourced from the xlsx share price files and Information Document, not the core PDFs)
`share_price_close`, `share_price_open`, `share_price_high`, `share_price_low`, `share_price_volume`, `admission_price`, `share_option_pool_percentage`

## 8. Error Handling

- **Malformed JSON from an extraction call:** retry once with the parse error fed back to the model ("your last response wasn't valid JSON: [error] — return valid JSON only"). If the retry also fails, the document is flagged `extraction_status: failed` rather than silently storing partial or garbage data, and the raw model response is logged for manual review.
- **Document has no risk section (e.g. a press release):** the risk extraction call is skipped entirely for that document type — this is a normal, expected case, not an error condition.
- **No prior risk records exist yet to diff against** (first document ever processed): the status-diffing call is skipped, and every extracted risk is set to `status: "New"` by default, since there's nothing to compare against.

## 9. Pipeline Ordering for Derived Metrics

Within a single document-processing run: raw metric extraction runs first and is inserted into `disclosures`. Only once that insert succeeds does the deterministic derived-metrics function run, reading the just-inserted raw rows for that same document/period and inserting the corresponding `derived: true` rows. Both inserts happen inside one database transaction per document, so a document is never left in a state with raw metrics stored but no corresponding derived ratios (or vice versa) — either the whole document's processing succeeds, or none of it is committed.

## 10. Derived Metric Taxonomy

Found missing during an assignment alignment check — Bucket 3/4 defined *how* derived metrics get computed (in code, from raw rows) but never listed *which* ones, even though several are named explicitly in the brief.

| Derived metric | Formula |
|---|---|
| `gross_margin` | gross_profit ÷ revenue |
| `operating_margin` | operating_profit_loss ÷ revenue |
| `total_assets` | (goodwill + development_costs + tangible_assets) + (debtors + cash_and_equivalents) — a subtotal, not disclosed as a single line item, needed as an input to ROCE |
| `ebitda` | operating_profit_loss + depreciation_amortization |
| `ebitda_margin` | ebitda ÷ revenue |
| `working_capital` | (debtors + cash_and_equivalents) − creditors_due_within_one_year |
| `cash_runway_months` | cash_and_equivalents ÷ average monthly net cash used in operations |
| `roce` | operating_profit_loss ÷ (total_assets − creditors_due_within_one_year) |
| `ebitda_to_fcf_bridge` | Not a single derived number — a waterfall presentation of ebitda → net_cash_operating_activities → net_cash_investing_activities → net change in cash. A chart/UI design decision (Bucket 8), not an extraction concern. |
